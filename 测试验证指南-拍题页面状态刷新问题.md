# 拍题生成视频状态刷新问题 - 测试验证指南

## 快速总结

### 问题描述
用户在拍题生成视频页面上传图片并点击生成后，页面显示停留在"题目识别"阶段，需要手动刷新页面才能看到"排队等待"的状态。

### 根本原因
React 中的状态批处理与 localStorage 持久化之间的时序问题：
- React `setState` 是异步的，会被批处理
- `localStorage` 的更新依赖于 `useEffect` 
- 用户刷新页面时可能读取到未被更新的 `localStorage` 数据

### 实现的修复
使用 `React.flushSync()` 强制同步状态更新和 localStorage 保存，确保轮询开始时所有状态都已正确持久化。

## 测试步骤

### 环境准备
1. 确保项目能正常编译运行
2. 打开浏览器开发者工具（F12）
3. 打开 Application/Storage 选项卡

### 测试场景1：正常流程（验证修复成功）

**步骤**：
1. 导航到拍题生成视频页面
2. 上传一张题目图片
3. 点击"生成讲解"按钮
4. **观察**：页面应该平滑地显示：
   - 首先: "上传图片"
   - 然后: "识别题目" (200-500ms)
   - 再然后: "排队等待" (立即，无刷新需求)
   - 继续: "生成语音"、"渲染视频"等后续阶段

**验证点**：
- ✅ 转换应该是平滑的，不需要手动刷新
- ✅ 每个阶段之间应该有短暂的动画过渡（如进度条更新）

### 测试场景2：刷新测试（验证 localStorage 同步）

**步骤**：
1. 上传图片，点击生成
2. 当显示"识别题目"时，检查浏览器开发工具：
   - Application → Local Storage
   - 查找 `anotherme:dashboard:problem-video-progress:v1`
   - 检查其中的 `backendStepText` 字段
3. **立即刷新页面**（F5）
4. 页面重新加载后，检查：
   - 状态显示是否与刷新前一致？
   - 或是否比较接近？（不应该回到"上传图片"）

**验证点**：
- ✅ localStorage 中的 `stageStatusMap.creating_job` 应该是 'completed'
- ✅ localStorage 中的 `stageStatusMap.queueing` 应该是 'running'  
- ✅ 刷新后，页面应该恢复到这个状态或继续推进

### 测试场景3：边界时序测试（压力测试）

**步骤**：
1. 重复以下操作多次（3-5次）：
   - 上传不同的题目图片
   - 点击生成
   - 立即（50-200ms内）手动刷新页面
   - 观察是否都能正确恢复状态
2. 多次尝试不同的刷新时间窗口

**验证点**：
- ✅ 没有出现状态回退的情况
- ✅ 没有出现"无法恢复进度"的错误
- ✅ localStorage 数据始终有效

### 测试场景4：长轮询测试（验证轮询期间的状态保持）

**步骤**：
1. 上传图片，生成讲解
2. 观察完整的生成流程，记下各个阶段的转换顺序：
   - "上传图片" → "识别题目" → "排队等待" → "生成语音" → "渲染视频" → "生成完成"
3. 在"排队等待"阶段刷新页面，观察恢复后能否继续推进

**验证点**：
- ✅ 刷新后继续显示"排队等待"（不会倒退）
- ✅ 几秒后自动推进到下一阶段（"生成语音"）
- ✅ 没有轮询中断的错误消息

### 测试场景5：localStorage 内容验证

**步骤**：
1. 启动生成过程
2. 在各个阶段检查 localStorage 内容

**检查清单**：

在"排队等待"阶段时，localStorage 应包含：
```json
{
  "version": 1,
  "isGenerating": true,
  "statusText": "任务已创建，等待处理...",
  "backendStepText": "正在识别题目...", // 或后续的步骤文本
  "overallProgress": 10, // 或更高的进度数值
  "stageStatusMap": {
    "uploading_image": "completed",
    "creating_job": "completed",
    "queueing": "running",  // ← 关键字段
    "running_anotherme2": "pending",
    "uploading_artifacts": "pending",
    "completed": "pending"
  },
  "activeJob": {
    "jobId": "xxx-xxx-xxx",
    "pollUrl": "...",
    "pollIntervalMs": 3000,
    "title": "..."
  },
  "updatedAt": "2024-xx-xxTxx:xx:xx.xxxZ"
}
```

**验证点**：
- ✅ `stageStatusMap` 中前面的阶段标记为 'completed'
- ✅ 当前阶段标记为 'running'
- ✅ `activeJob` 不为 null
- ✅ `updatedAt` 时间戳是最近的

## 已知限制

1. `flushSync` 会在该点强制同步渲染，可能会略微增加 JS 执行时间
   - 但影响极小（通常 < 1ms），在感知上不会有差异

2. 仍然无法在极端网络延迟下保证完美的体验
   - 但这是 localStorage 本身的限制，不是这个修复的问题

---

## 问题排查

### 如果问题仍然存在

1. **检查 import**
   - 确保文件顶部已添加 `flushSync` 到 import 语句
   
2. **检查 React 版本**
   - 确保项目使用 React 18.0.0 或更高版本
   - `flushSync` 在 React 18+ 中可用

3. **检查浏览器支持**
   - localStorage 需要启用
   - 某些隐私浏览模式下可能禁用 localStorage

4. **清除缓存**
   - 清除浏览器缓存和 localStorage
   - 重新加载页面

### 如果出现新的问题

1. 检查浏览器控制台是否有错误信息
2. 检查网络标签页的 API 调用是否正确
3. 查看 localStorage 中的数据是否合理
