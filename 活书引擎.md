# AnotherMe 活书引擎对齐 DeepTutor 优化计划

## 对比结论
- AnotherMe 已有活书骨架：多块类型、SSE 进度、块级错误、健康检查、自动编译未就绪页都已存在，核心在 [live-book-store.ts](D:/AnotherMe-main/AnotherMe/features/live-book/server/live-book-store.ts:781) 和 [live-book-compiler.ts](D:/AnotherMe-main/AnotherMe/features/live-book/server/live-book/live-book-compiler.ts:515)。
- 主要差距在生成质量链路：DeepTutor 是“输入捕获 → LLM 提案 → 多源探索 → 书脊批判修订 → LLM 页面规划 → 块级生成器”，AnotherMe 目前多为确定性模板和主题拼装。
- 最大问题不是 UI 缺按钮，而是内容证据不足、书脊不够智能、块内容多为占位/静态文案，导致“像活书界面，但不像真正资料驱动的活书”。

## 关键差距
- 输入与证据：AnotherMe 创建接口只收 `topic/language/targetLevel/sources.text`，SourceExplorer 主要从主题、章节、错题和手填 sources 拼 chunks；DeepTutor 的 BookCreator 支持 KB、笔记、题库、聊天选择，并由 SourceExplorer 做 LLM query design + 并行 RAG + notebook/chat 聚合，见 [source_explorer.py](D:/AnotherMe-main/DeepTutor/deeptutor/book/agents/source_explorer.py:97)。
- 书脊质量：AnotherMe 的 [spine-synthesizer.ts](D:/AnotherMe-main/AnotherMe/features/live-book/server/live-book/spine-synthesizer.ts:161) 是候选概念模板化生成；DeepTutor 的 [spine_synthesizer.py](D:/AnotherMe-main/DeepTutor/deeptutor/book/agents/spine_synthesizer.py:82) 有 Draft → Critique → Revise、拓扑排序、循环移除和覆盖补齐。
- 页面与块生成：AnotherMe 的 BlockGeneratorRegistry 目前多数生成静态内容；DeepTutor 的 [page_planner.py](D:/AnotherMe-main/DeepTutor/deeptutor/book/agents/page_planner.py:256) 先用 LLM 规划块序列，块生成器再结合 exploration/RAG 产出结构化 payload。
- 编译调度：AnotherMe 有事件和重连，但 `runCompileWorkflow` 仍是单流程顺序编译；DeepTutor 有 per-book queue、当前页优先、后台补齐、逐块持久化，见 [engine.py](D:/AnotherMe-main/DeepTutor/deeptutor/book/engine.py:14)。
- 阅读器成熟度：AnotherMe 的读者页集中在一个大组件内；DeepTutor 拆成 PageReader、BlockRenderer 和各块组件，并支持错误 Retry、换类型、插入、移动等更完整编辑体验，见 [BlockRenderer.tsx](D:/AnotherMe-main/DeepTutor/web/app/(workspace)/book/components/blocks/BlockRenderer.tsx:62)。

## 优化方案
- P0：把 UI 的确认流程改为先调用现有 `/api/live-book/confirm-proposal`，再确认书脊，确保 SourceExplorer + SpineSynthesizer 在用户确认阶段真实生效，而不是只在 compile 时后台改目录。
- P0：扩展 `CreateLiveBookInput.sources` 为结构化引用：`kbIds/notebookRefs/chatSelections/questionRefs/manualText`，保留当前 `sources.text` 兼容字段；SourceExplorer 改为先接真实 KB/RAG、笔记、聊天、题库查询。
- P1：把 SpineSynthesizer 改为 LLM-first、模板 fallback：输出 `chapters + conceptGraph + critiqueIssues`，并做 schema 校验、去重、依赖排序和 overview chapter 注入。
- P1：引入 SectionArchitect：按章节 `contentType` 和 exploration summary 规划块序列；现有 Registry 保留为 fallback，但 section/quiz/figure/flash_cards/code/timeline 至少应生成真实结构化 payload。
- P2：改编译调度为页优先队列：用户打开页立即编译，后台继续补齐剩余页；每个 block 完成即保存并发事件，失败块不阻断整页。
- P2：前端拆分 `features/live-book/pages/live-book/page.tsx`：保留工作台壳，抽出 Creator、SpineEditor、Reader、ProgressTimeline、HealthBanner、BlockRenderer，减少后续迭代风险。

## 测试与验收
- 单元测试：SourceExplorer 输入选择、去重、coverage；SpineSynthesizer schema/fallback/依赖排序；SectionArchitect 模板 fallback；block 失败后 page 为 `partial`。
- API 测试：create → confirm-proposal → confirm-spine → compile-page → regenerate-block → health → refresh-fingerprints 全链路。
- 前端验收：选择 KB/笔记/聊天/题库创建活书；确认提案后目录能基于资料变化；打开未编译页自动生成；错误块可重试；健康横幅可重编 stale 页。
- 质量验收：生成页面必须显示 source anchors，至少 section/quiz/callout/concept_graph 不再是纯模板文案。

## 假设
- DeepTutor 作为参考实现，不直接作为 AnotherMe 运行时依赖。
- 当前工作区有大量未提交改动，本计划基于当前文件状态，不要求回滚或重构无关模块。
- 优先做“按能力迁移”，不照搬 Python 后端；AnotherMe 继续用现有 Next.js/TypeScript 服务端架构。
