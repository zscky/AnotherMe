# 拍题生成视频页面需要刷新才能从"题目识别"转到"排队等待"的问题分析

## 问题现象
用户在拍题生成视频页面点击生成后：
1. 页面显示"识别题目"阶段（创建任务中）
2. 后端已经返回了任务ID，页面应该转入"排队等待"阶段
3. 但需要手动刷新页面，UI才会更新为"排队等待"状态

## 根本原因分析

### 关键代码位置
文件：`app/(dashboard)/photo-to-video/page.tsx`

#### 问题1：React setState 批处理时序问题
在 `handleGenerate()` 中（第 820-870 行）：

```typescript
// 第一批 setState
markStageAsActive('creating_job');        // 内存状态 → 'creating_job'
setOverallProgress((prev) => Math.max(prev, 8));

await sleep(150);  // ← 关键：这里 await 可能导致批处理被拆分

// 第二批 setState（排队状态）
markStageAsActive('queueing');            // 内存状态 → 'queueing'
setStatusText('任务已创建，等待处理...');
setOverallProgress((prev) => Math.max(prev, 10));
setActiveJob(createdJobMeta);

await pollJobUntilTerminal({...});  // ← 立即开始轮询
```

**问题**：
- `await sleep(150)` 后的 setState 会被 React 批处理
- 但轮询循环在同步代码中会立即启动
- 如果用户在 useEffect 保存 localStorage 之前刷新，会看到过时的状态

#### 问题2：轮询过程中缺少状态同步
在 `pollJobUntilTerminal()` 中（第 676-760 行）：

```typescript
applyBackendProgress(pollPayload.step, pollPayload.progress);  // ← 更新内存状态

setStatusText(
  `${BACKEND_STEP_LABEL[...] || ...}${progressText}`,  // ← 更新 UI 状态
);

// 但 setStageStatusMap 的更新可能滞后于轮询的快速迭代
```

**问题**：
- 轮询循环会频繁调用 setState
- 这些 setState 会被批处理
- 但如果轮询间隔很短（默认3秒），可能会有多个 setState 批处理待处理
- 期间用户刷新，会读取到过时的 localStorage

#### 问题3：localStorage 保存时序问题
useEffect 中（第 391-405 行）：

```typescript
useEffect(() => {
  if (!hasRestoredProgress) return;
  saveProgressSnapshot({
    version: PROGRESS_SNAPSHOT_VERSION,
    isGenerating,
    statusText,
    backendStepText,
    overallProgress,
    stageStatusMap: normalizeStageStatusMap(stageStatusMap),
    activeJob,
    updatedAt: new Date().toISOString(),
  });
}, [
  hasRestoredProgress,
  isGenerating,
  statusText,
  backendStepText,
  overallProgress,
  stageStatusMap,
  activeJob,
  // ← 依赖项很多，任何一个变化都会触发
]);
```

**问题**：
- useEffect 依赖于多个状态变量
- 当这些变量变化时，useEffect 会被调用来保存 localStorage
- 但在 React batch 完成和 useEffect 运行之间，可能有一个时间窗口
- 如果用户在这个窗口内刷新，会看到不一致的状态

## 具体场景重现

1. **T0**: 用户点击"生成"
2. **T0+0ms**: `handleGenerate()` 开始执行
3. **T0+5ms**: 创建任务成功，后端返回 `jobId`
4. **T0+10ms**: 调用 `markStageAsActive('creating_job')` → setState #1
5. **T0+12ms**: 调用 `setOverallProgress(8)` → setState #2
6. **T0+15ms**: `await sleep(150)` 等待...
7. **T0+165ms**: 调用 `markStageAsActive('queueing')` → setState #3
8. **T0+167ms**: 调用 `setStatusText(...)` → setState #4
9. **T0+169ms**: 调用 `setOverallProgress(10)` → setState #5
10. **T0+171ms**: 调用 `setActiveJob(...)` → setState #6
11. **T0+172ms**: **← 用户此时刷新页面**
    - React 还没完成批处理
    - useEffect 还没保存 localStorage
    - 或者虽然 setState #6 已提交，但 React 渲染还没完成
    - 读取 localStorage 得到旧的状态值（still in 'creating_job'）
12. **T0+180ms**: React 完成批处理 #3-#6，进行渲染
13. **T0+185ms**: useEffect 被调用，保存 localStorage（已经太晚）

## 解决方案

### 方案1：确保状态变化被立即同步化（推荐）
在调用 `pollJobUntilTerminal` 之前，加入一个微任务同步点：

```typescript
// 在 handleGenerate 中，pollJobUntilTerminal 前添加：
setActiveJob(createdJobMeta);

// ← 关键改进：让 React 有机会完成批处理和 useEffect
await new Promise(resolve => {
  setTimeout(resolve, 0);  // 让出事件循环一个周期
});

await pollJobUntilTerminal({...});
```

### 方案2：在轮询循环中主动保存状态快照
在 `pollJobUntilTerminal` 中，每次状态更新后，主动保存：

```typescript
applyBackendProgress(pollPayload.step, pollPayload.progress);
setStatusText(...);

// ← 关键改进：主动保存而不依赖 useEffect
saveProgressSnapshot({
  version: PROGRESS_SNAPSHOT_VERSION,
  isGenerating: true,
  statusText: ...,
  backendStepText: ...,
  overallProgress: ...,
  stageStatusMap: normalizeStageStatusMap(...),
  activeJob: jobMeta,
  updatedAt: new Date().toISOString(),
});
```

### 方案3：使用 React.flushSync 强制同步更新（最彻底）
```typescript
import { flushSync } from 'react';

// 在状态转移的关键点使用 flushSync
flushSync(() => {
  markStageAsActive('queueing');
  setStatusText('任务已创建，等待处理...');
  setOverallProgress(10);
  setActiveJob(createdJobMeta);
});

// 此时所有状态已同步更新并渲染完成
await pollJobUntilTerminal({...});
```

## 建议的修复步骤

1. 立即应用**方案1**（最小改动，安全可靠）
2. 可选地应用**方案2**（提高鲁棒性）
3. 监控修复效果，确认问题消除
